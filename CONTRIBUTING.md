# Contributing to Bahrain FNSL: Join the Fight

Look, this isn't just some academic exercise. This is about real people, real health, and real impact in Bahrain. If you're a researcher, a clinician, a developer, or just someone who gives a damn about this mission, we need you. This isn't a polished corporate project; it's a movement. Here's how you can jump in.

## First Rule: Don't Be a Jerk

Seriously. Read our `CODE_OF_CONDUCT.md`. We're building something important here, and we need a community that's respectful, inclusive, and focused. No time for drama.

## How to Actually Help

### Found a Bug? Tell Us.

If something's broken, don't just grumble. Open an issue on GitHub. Make it clear:
-   What broke.
-   How you broke it (steps to reproduce).
-   What you expected to happen vs. what actually happened.
-   Your setup (Python version, OS, etc.).

### Got an Idea? Shout It Out.

Think you can make this better? We're all ears. Open an issue. Don't be shy:
-   What's your idea?
-   Why does it matter? What problem does it solve?
-   (Optional) How would you even start to build it?

### Want to Code? Prove It.

This is where the rubber meets the road. If you want to contribute code:

1.  **Fork it.** Get your own copy.
2.  **Branch out.** `git checkout -b feature/your-awesome-feature`
3.  **Code it.** Make your changes. Make them good. Test your stuff. Seriously, test it.
4.  **Commit it.** Write clear, concise commit messages. Don't just say "fixed stuff."
5.  **Push it.** `git push origin feature/your-awesome-feature`
6.  **PR it.** Open a Pull Request. Explain what you did and why. We'll review it. We'll probably have questions. Be ready.

## The Nitty-Gritty (Dev Guidelines)

### Code Style

-   PEP 8. Just follow it. No excuses.
-   Meaningful names. Don't make us guess what `x` or `temp_var` does.
-   Docstrings. Explain your functions and classes. Assume the next person reading your code is tired and confused.
-   Keep it tight. Functions should do one thing, and do it well.

### Testing

-   If you add a feature, add tests for it. Period.
-   All tests must pass. No exceptions.
-   Aim for decent code coverage. We're not looking for 100%, but don't ignore the basics.

### Docs

-   If you change something, update the docs. `README`, `GETTING_STARTED`, whatever's relevant.
-   Examples. Show us how it works.

## Where We Need Help (High-Impact Areas)

### Urgent Stuff

-   **Real Data Integration**: This is the big one. Connecting to actual hospital data (securely, obviously).
-   **Non-IID Optimization**: Our federated models need to be robust when data across hospitals is wildly different.
-   **Next-Gen Privacy**: Homomorphic encryption, secure enclaves. Let's push the boundaries.
-   **Deployment Guides**: How do we get this into a real clinic? We need practical guides.

### Important Stuff

-   **More Diseases**: T2DM is just the start. Let's tackle SCD crises, hypertension, you name it.
-   **Better Visuals**: Make the plots even more intuitive. Help doctors *see* the impact.
-   **Performance Benchmarking**: How fast is it? How scalable? We need numbers.

### Good to Have Stuff

-   UI/UX. Make it pretty. Make it easy.
-   More examples, more tutorials. Help others get started.
-   Refactoring. Always refactoring. Make the code cleaner, faster, stronger.

## Questions? Problems? Just Ask.

Open a GitHub Discussion. Or hit us up directly. We're all in this together.
